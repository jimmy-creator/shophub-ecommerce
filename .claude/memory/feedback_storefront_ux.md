Several non-obvious storefront patterns were established across this codebase. These are easy to accidentally re-break because they involve "remove the obvious thing" or "don't use the project's standard token here":

1. **Cart-added popup is the canonical add-to-cart feedback — never add toasts.** `<CartAddedPopup>` is rendered globally inside `CartProvider` (in `client/src/context/CartContext.jsx`); it consumes `lastAdded` (set on every `addToCart` call) and shows a top-right card with the product, qty, and View Cart / Continue buttons. When wiring a new "add to cart" path, **do not** add `toast.success('Added to cart')` next to the `addToCart(...)` call — that creates a double notification. The seven legacy toast calls were already removed across `ProductCard`, `ProductDetail`, and `Wishlist` for all three stores.

2. **Mobile inputs must be ≥16px or iOS Safari auto-zooms on focus.** `client/src/index.css` has a `@media (max-width: 720px)` block at the bottom that forces every text-style `input/select/textarea` to `font-size: 16px !important`. New form components do not need to think about this — but **do not override** font-size below 16px in a higher-specificity rule scoped to mobile, or the zoom will return.

3. **UI primitive chrome must not depend on theme tokens.** The cart popup originally used `var(--radius-lg)` for its card and `var(--radius)` for buttons. On the VPS, the active theme falls back to `default` (whose `vars: {}` is empty), so the base `--radius-lg: 8px` was used and the popup looked under-rounded. Fixed by hardcoding `18px / 12px / 10px`. **Pattern:** any UI primitive that should look identical regardless of which theme an admin selects (popups, modals, toasts, dropdowns) should hardcode its radii / shadows. Only theme-driven surfaces (cards, buttons inside content) should use theme tokens.

4. **No `backdrop-filter` on iOS mobile.** Store3's `layout.css` has a mobile-only block that disables `backdrop-filter` on `.s2-glass`, `.s2-nav`, `.s2-bottom-nav.s2-bottom-nav-always`, `.s2-pricing`, `.s2-banners`, `.s2-banner-slide`, and search dropdowns, plus hides `.s2-root::before/::after` (the animated aurora gradient + grain SVG). Reason: combined GPU/memory pressure caused iOS Safari to crash with "A problem repeatedly occurred" on pinch-zoom. **Don't reintroduce `backdrop-filter: blur()` on sticky / fixed elements in store3** — use solid backgrounds.

5. **`--s2-lavender` is not lavender.** Despite the name (inherited from store2), `--s2-lavender` and `--s2-lavender-deep` are the **primary accent color** for store2/store3 — used in 36+ places (italic emphasis, primary buttons, active nav, focus borders, gradients). Store2 sets it to crimson (`#8B1A1A`); store3 sets it to near-black (`#171311 / #000000`). When changing the accent for a store, change those two variables; do not hunt for individual usages.

6. **Admin product list fetches `?limit=10000`.** `GET /products/admin/all` defaults to a high limit (was 200 → bumped). The five admin client fetch sites in `client/src/pages/Admin.jsx` all pass `?limit=10000` explicitly. If you add a new admin product fetch, **pass the explicit limit** so the panel doesn't silently truncate after a CSV import grows the catalog past the default.

7. **CSS Grid columns: always use `minmax(0, 1fr)` not `1fr`.** Plain `1fr` is implicitly `minmax(auto, 1fr)` so tracks expand to fit min-content. With an `<img width="400">` (or any wide content) inside a card, this blew the products grid out so each cell was 400px+ wide instead of the intended ~218px (1/4 row). Affected both `.s2-grid` and `.s2-products-layout` columns at every breakpoint. Same `min-width: 0` applied to `.s2-product` so the article itself can shrink below its child img's intrinsic width. **Also: don't add `width="400" height="400"` attributes to product card `<img>`s** — `.s2-product-img` already has `aspect-ratio: 1` to reserve space, the attribute creates a 400px min-content trap.

8. **Pagination pattern is windowed Prev/Next, not all-N-buttons.** All three stores (`Products.jsx` in `store1/`, `store2/`, `store3/`) render a 10-page sliding window centered on the current page, plus `‹` and `Next ›` buttons. On mobile (`@media (max-width: 720px)`), CSS hides the numbered buttons and any disabled nav arrow, and shows a `<span class="(s2-)pagination-info">Page X of N</span>` instead. The fetch effect calls `window.scrollTo({ top: 0, behavior: 'smooth' })` on every `[page, category, search, sort, order]` change so users land at the top after navigation. Original bug: rendering `Array.from({length: totalPages})` produced 100+ buttons that wrapped across the viewport.

9. **More misleadingly-named tokens in store2/store3 layout.css.** Discovered alongside `--s2-lavender` (which is red/black). `--s2-success` is green (`#16a34a`) and is the source of the "ADDED TO CART" header, in-stock badge, discount row, pincode-available feedback, admin "Paid" status, and ~20 other places. `--s2-rose` and `--s2-rose-deep` are **teal** (`#14b8a6 / #0d9488`), used by `.s2-badge` (cart count badge), `.s2-bottom-nav-badge`, and a gradient at `layout.css:1465`. When rebranding a store's accent, swap **all three pairs** in one go: `--s2-lavender(-deep)` (primary accent), `--success` / `--s2-success` (success/added), and `--s2-rose(-deep)` (count badges). Store3's Kalif rebrand uses `#3B277E` purple for all three.

10. **Admin content-types-as-settings pattern.** New admin features (announcements, category cards, mid-banners) all use the same shape: one row in the `Settings` table keyed by the feature name (e.g. `announcements`, `category-cards`), with `value` storing a JSON array. Backend pattern: `GET /settings/<key>` (public) returns the array, `PUT /settings/<key>` (admin) replaces it with length validation. Admin pattern: dedicated editor component (`AnnouncementEditor`, `CategoryCardsEditor`, or `BannerEditor` with `endpoint`/`title`/`maxBanners` props) that reads, edits in local state, and saves on field blur / button click. Storefront pattern: fetch on mount + localStorage cache so the next page load is instant. **When adding a new admin-managed content type, copy this triple — don't invent a new shape.**

11. **react-icons fa6 simplified the brand-icon names.** `react-icons/fa6` (FontAwesome 6) renamed some icons compared to `react-icons/fa` (FA5). Notably `FaTelegramPlane` (FA5) → `FaTelegram` (FA6). Most others kept their names. If a build dies with `"FaXxx" is not exported by ".../react-icons/fa6/..."`, drop the suffix or check the FA6 list before assuming the icon is missing.

12. **Mobile "drop search to second row" pattern.** When the navbar gets too crowded on mobile (hamburger + logo + icons + search), the search wrapper takes `flex-basis: 100%` + `order: 10` on its parent `.s2-nav { flex-wrap: wrap; row-gap: 0.6rem; }`. The other items stay on row 1 in their natural order; the search wraps to row 2 because of the 100% basis, and `order: 10` keeps it last even if any items are added after it in the JSX. This is the standard mobile pattern across store3's restructured Farmley-style navbar.

**Why these matter:** every one of these has been a bug a real user hit, and they're invisible to grep — adding a toast next to `addToCart` "looks right", `var(--radius-lg)` "looks right", `font-size: 0.95rem` on a search input "looks right". The git history will show when each was introduced/fixed but won't stop the next regression.

**How to apply:** When touching any cart/checkout/popup/modal code, search this memory for the relevant pattern before writing. When changing input font-size on mobile, double-check it stays ≥16px. When adding a sticky/fixed element with frosted glass on store3, scope the `backdrop-filter` to `min-width: 721px` only.
